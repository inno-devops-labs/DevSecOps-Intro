### What was done
1. Pulled target image `bkimminich/juice-shop:v19.0.0`.
2. Started local registry container `registry:3` mapped to `localhost:5001`.
3. Pushed image to local registry as `localhost:5001/juice-shop:v19.0.0`.
4. Resolved digest and used digest reference for signing:

   - New digest became `sha256:c4e5b27bf840ba1ebd5568b6b914f6926f3559b2ad4f505b1f37aae483b907d6`.
   - Verification for tampered digest fails (`no signatures found`).

### Analysis
- **How signing protects against tag tampering:**
  signature is bound to image content digest, not tag name. If attacker retargets `:v19.0.0` to different content, digest changes and existing signature no longer matches.
- **What â€śsubject digestâ€ť means:**
  immutable cryptographic hash of the exact artifact (image manifest/content) that signature or attestation refers to.

### Analysis
- **Signature vs attestation:**
  signature answers â€śis this artifact signed by trusted key?â€ť, while attestation adds signed metadata claims about artifact (e.g., SBOM/provenance).
- **What SBOM attestation contains:**
  signed software inventory metadata for the subject image: components/packages, versions, dependency graph, and metadata generated by Syft/CycloneDX.
- **What provenance attestation provides:**
  signed build context (builder identity, build type, invocation parameters, timestamp/completeness). This supports traceability and policy checks in supply chain controls.

- Blob verify output: `labs/lab8/artifacts/verify-blob.txt`

### Analysis
- **Use cases for non-container artifact signing:**
  release binaries, scripts, configuration files, IaC templates, policy bundles, offline update packages.
- **How blob signing differs from image signing:**
  blob signing targets a file payload directly (local path/hash) and can use detached/bundle verification, while image signing targets OCI image digest in a registry and often includes registry/referrer workflows.
