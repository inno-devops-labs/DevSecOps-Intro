## Task 2: Trivy Comparison

### Side-by-side counts

| Severity | Grype | Trivy | Delta |
|----------|------:|------:|------:|
| Critical | 7 | 5 | -2 |
| High | 51 | 43 | -8 |
| Medium | 35 | 39 | 4 |
| Low | 4 | 22 | 18 |
| **Total** | 97 | 109 | 12 |

Grype also reported 7 Negligible findings. Trivy was run with `--severity LOW,MEDIUM,HIGH,CRITICAL`, so the side-by-side table compares the shared severity bands only.

### Why the difference?

1. `CVE-2015-9235` was reported by Trivy for `jsonwebtoken` 0.1.0 and 0.4.0, while Grype reported the same affected package/version as `GHSA-c7hr-j4mj-j2w6`. This is an advisory alias-normalization difference: strict ID comparison makes it look like Grype missed the CVE, but Grype is using the GitHub Security Advisory ID as the primary identifier.

2. `CVE-2019-10744` was reported by Trivy for `lodash` 2.4.2, while Grype reported the matching critical lodash advisory as `GHSA-jf85-cpcp-j695`. This again shows different vulnerability database and identifier choices; for real triage I would normalize aliases before claiming that one scanner truly missed the vulnerability.

### When would you pick each?

Syft + Grype's decoupled model wins when the SBOM itself is a durable artifact: one inventory can be stored, signed, attached to a release, and rescanned later when vulnerability intelligence changes. That is the better fit for Lab 8-style attestations and incident response questions such as "did this shipped image contain package X?"

Trivy's all-in-one model wins when I need a simple CI job with broad coverage and low setup cost. It can scan image vulnerabilities, IaC, secrets, and misconfiguration in one tool, which is useful for quick gates even if it gives less explicit separation between inventory generation and vulnerability evaluation.
